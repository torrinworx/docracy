# Phase 8: Workspace Tenancy via MCP Session Binding and Postgres RLS Isolation - Research

**Researched:** 2026-04-08
**Domain:** Workspace-scoped MCP sessions with Postgres RLS isolation
**Confidence:** MEDIUM

<user_constraints>
## Locked Decisions
- Workspace identity must be generated, not derived from folder names or working-directory heuristics.
- The MCP client must be able to bind a workspace explicitly through project-scoped config/environment.
- Global governance stays shared; the tenancy model must not duplicate repo-owned instructions per workspace.
- Postgres RLS must enforce isolation for documents, revisions, and raw SQL.

### the agent's Discretion
- Whether the shared scope is represented by a reserved global workspace row or a nullable workspace id.
- Exact helper names and config field placement, so long as the process binds once at startup and keeps the same session scope.
- Whether workspace binding is optional fallback for the shared/global path or required only when a workspace-aware client is used.

### Deferred Ideas
- Multi-user auth, OAuth, or per-workspace RBAC.
- A dedicated workspace management UI or extra tool surface.
- Vector search, branching, or other retrieval changes.

</user_constraints>

<research_summary>
## Summary

The cleanest fit for Docracy is a tenant boundary built on a reserved global scope plus a session-scoped custom GUC such as `docracy.workspace_id`. The MCP process should bind that session value once at startup and keep it for the process lifetime so every query, read, write, and raw SQL call flows through the same Postgres RLS policy.

OpenCode already supports project-scoped `opencode.json` and environment injection for MCP servers, so the repository can ship a checked-in example that passes `WORKSPACE_ID` through the client config without inventing a new local binding file. The workspace id itself should be a UUID generated by the system, while shared governance lives in the global scope so the existing init path does not lose access to repo-owned instructions.

</research_summary>

<standard_stack>
## Standard Stack

| Component | Why |
|-----------|-----|
| PostgreSQL RLS | Enforces workspace isolation at the database layer instead of relying on caller discipline. |
| Custom GUC (`current_setting` / `set_config`) | Lets the server bind workspace context once per session/process and reuse it in policies. |
| SQLx pool `after_connect` or startup session setup | Ensures every pooled connection carries the same workspace context. |
| OpenCode project config | Provides a checked-in, per-project MCP environment block for `WORKSPACE_ID`. |

### Recommended Query/RLS Pattern

- Keep `documents` and `document_revisions` keyed by `workspace_id`.
- Allow the current workspace plus a reserved global workspace in RLS policies.
- Keep raw SQL on the same tables and session so the policy applies automatically.

</standard_stack>

<architecture_patterns>
## Architecture Patterns

### Pattern 1: Session-bound tenant context
**What:** Bind `docracy.workspace_id` at startup and let the database enforce the boundary.
**Why:** It keeps scope out of business logic and makes raw SQL obey the same isolation rules.

### Pattern 2: Shared/global plus workspace-specific rows
**What:** Keep repo-owned governance in the global scope and user/workspace documents in their own workspace rows.
**Why:** Existing init and governance behavior stays usable while tenant isolation is added.

### Pattern 3: Project config as the binding source
**What:** Put the workspace id in project-scoped MCP config/environment, not in a Docracy-managed local binding file.
**Why:** The client already owns process startup; the server should only validate and consume the binding.

### Anti-Patterns to Avoid
- Deriving workspace identity from folders, repo names, or paths.
- Letting pooled connections inherit stale tenant state from prior requests.
- Creating a second Docracy-specific binding file when the client config already supports environment injection.

</architecture_patterns>

<common_pitfalls>
## Common Pitfalls

### Pitfall 1: Connection pool leakage
**What goes wrong:** A connection is reused with the wrong workspace still set.
**How to avoid:** Bind the workspace during connection setup or checkout, not only in one handler.

### Pitfall 2: RLS bypass by privileged roles
**What goes wrong:** Owner/bypass roles silently skip policies.
**How to avoid:** Keep the app role ordinary and rely on RLS policies rather than privileged ownership for tenant checks.

### Pitfall 3: Global governance becomes inaccessible
**What goes wrong:** Binding a workspace hides the shared repo-owned instructions needed by Init.
**How to avoid:** Keep a reserved global scope and include it in the policy.

### Pitfall 4: Raw SQL bypasses the tenant boundary
**What goes wrong:** The direct SQL path reads the whole table because it is not using the same session context.
**How to avoid:** Make raw SQL go through the same scoped repository/session as the normal query path.

</common_pitfalls>

<open_questions>
## Open Questions

1. **Reserved global workspace row vs nullable workspace id?**
   - Recommendation: use a reserved global workspace row so foreign keys and policies stay explicit.

2. **Should missing `WORKSPACE_ID` be fatal?**
   - Recommendation: no for shared/global mode, yes for workspace-bound client configs that expect isolation.

3. **Should workspace ids use UUIDv4 or UUIDv7?**
   - Recommendation: use the existing generated UUID style that is already wired into the workspace code path unless a later phase wants sortable ids.

</open_questions>
