Split the core query contract so `query` is strictly Postgres-backed (guided + raw SQL), and introduce a separate, explicit-embedding vector query contract + use-case that interface layers can call after generating embeddings.

Output: Postgres-only `QueryInput`, new `QueryVectorInput`, new `query_vector_documents` core use-case, and updated core tests.
</objective>

  </behavior>
  <action>
    In `crates/core/src/query.rs`, add a new serde-friendly input contract:
    - `pub struct QueryVectorInput` with fields:
      - `embedding: Vec<f32>`
      - `where_` (serde rename "where", default map)
      - `select: Vec<String>` (default empty)
      - `limit: Option<u32>`
    - Add a `parse(self) -> ValidationResult<(DocumentQuery, Vec<SelectField>, Map<String, Value>, Vec<f32>)>` (or equivalent) that:
      - rejects empty `embedding`
      - calls `parse_guided_query(None, where_, vec![], select, limit, None)` to reuse existing filter defaults
      - forces `DocumentQuery.query` to `None` (vector query text filtering is not part of this core contract)

    In `crates/core/src/service.rs`:
    - Add `pub async fn query_vector_documents(repo: &dyn Repository, input: QueryVectorInput) -> Result<QueryResult, CoreError>`.
    - Implement:
      1) parse QueryVectorInput into `(DocumentQuery, select_fields, applied_where, embedding)`
      2) call `repo.query_vector_documents(query.clone(), embedding).await?`
      3) preserve ranked order by hydrating `ranked_ids` via `repo.get_documents(&ranked_ids)` and re-ordering by id (same approach as the current vector branch)
      4) return `QueryResult { rows: project_rows(...), total_count, applied_where, next_cursor: None }`

    - Re-export `QueryVectorInput`.
